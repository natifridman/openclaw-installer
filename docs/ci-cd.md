# CI/CD Pipeline

## Overview

The OpenClaw Installer uses GitHub Actions for continuous integration and deployment validation. All pull requests must pass automated checks before merging.

## Workflow: CI Validation

**File:** `.github/workflows/ci.yml`

**Triggers:**
- Push to `main` branch
- All pull requests
- Manual workflow dispatch

**Jobs:**

### Validate
Runs on Ubuntu latest with Node.js 22.

**Steps:**
1. **Checkout** - Fetches repository code
2. **Set up Node.js** - Installs Node.js 22 with npm cache
3. **Install dependencies** - Runs `npm ci` for clean install
4. **Lint** - Runs `npm run lint` (ESLint checks)
5. **Build** - Runs `npm run build` (vite + tsc compilation)
6. **Test** - Runs `npm test` (all 319 vitest tests)
7. **Publish Test Results** - Posts test results to PR checks
8. **Upload Test Results** - Archives test results as artifacts (30-day retention)

**Test Reporting:**
- JUnit XML format generated by vitest
- Test results visible in PR checks tab
- Pass/fail status shown in PR checks
- Test artifacts available for download

## Branch Protection Rules

To enforce CI validation and maintain code quality, configure the following branch protection rules for the `main` branch:

### Recommended Settings

Navigate to: **Repository Settings → Branches → Branch protection rules → Add rule**

**Branch name pattern:** `main`

**Protection settings:**
- ✅ **Require a pull request before merging**
  - ✅ Require approvals: 1
  - ✅ Dismiss stale pull request approvals when new commits are pushed
- ✅ **Require status checks to pass before merging**
  - ✅ Require branches to be up to date before merging
  - **Required status checks:**
    - `validate` (CI workflow job)
    - `Test Results` (test reporting check)
- ✅ **Require conversation resolution before merging**
- ✅ **Do not allow bypassing the above settings** (applies to administrators)

### Setting Up Branch Protection

1. Go to repository settings: `https://github.com/<org>/<repo>/settings/branches`
2. Click "Add rule" or edit existing `main` branch rule
3. Configure the settings listed above
4. Save the rule

**Note:** You need admin or maintain permissions on the repository to configure branch protection rules.

## Local Pre-commit Validation

Before pushing commits, always run the full CI validation locally:

```bash
npm run build    # Compiles server + installer provider plugins (catches type errors)
npm test         # Runs all vitest tests (server + client)
npm run lint     # ESLint checks
```

See [AGENTS.md](../AGENTS.md) for the full pre-commit checklist.

## Success Criteria

✅ All PRs automatically run build + test + lint  
✅ Test failures block PR merge  
✅ CI status visible in PR checks  
✅ Main branch protected with required checks  
✅ Test results reported and archived

## Troubleshooting

### Workflow Fails on Lint
- Run `npm run lint` locally to see detailed errors
- Fix linting issues or update ESLint config if needed

### Workflow Fails on Build
- Run `npm run build` locally to reproduce
- Check TypeScript errors (most important - catches type errors `--noEmit` may miss)
- Verify all dependencies are installed (`npm ci`)

### Workflow Fails on Test
- Run `npm test` locally to reproduce
- Check test output for specific failures
- View test artifacts in workflow run for detailed results

### Test Results Not Showing
- Verify `publish-unit-test-result-action` has permissions
- Check that `test-results.xml` is being generated
- Ensure workflow has `pull-requests: write` and `checks: write` permissions

## Maintenance

### Updating Node.js Version
Edit `.github/workflows/ci.yml`:
```yaml
- name: Set up Node.js
  uses: actions/setup-node@v6
  with:
    node-version: 22  # Update this version
```

### Adding New Validation Steps
Add new steps to the `validate` job in `.github/workflows/ci.yml`. Keep steps in the recommended order:
1. Checkout & setup
2. Install dependencies
3. Lint
4. Build
5. Test
6. Additional validation (if needed)

### Modifying Test Reporting
Edit the test step reporter configuration:
```yaml
- name: Test
  run: npm test -- --reporter=verbose --reporter=junit --outputFile.junit=test-results.xml
```
